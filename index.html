Step 1 — Chunking strategies
How do you split the text?
Chunking is one of the most consequential decisions in a vector DB pipeline. Too small and you lose context. Too large and the meaning gets diluted. Click each strategy to see how it splits an example review.
fixed_size
Split every N tokens regardless of sentence boundaries. Simple and predictable.
✓ Fast, simple   ✗ Cuts mid-sentence
sentence
Split at sentence boundaries. Each chunk is a complete thought.
✓ Preserves meaning   ✗ Variable size
sliding_window
Chunks overlap by N tokens so context isn't lost at boundaries.
✓ No lost context   ✗ Duplicate storage
semantic
An AI groups sentences by topic shift. Best quality, most expensive.
✓ Best coherence   ✗ Slow, complex

Step 2 — Embedding
Each chunk becomes a vector
An embedding model (e.g. text-embedding-3-large) reads each chunk and outputs a list of numbers — its coordinates in meaning-space. Semantically similar chunks will produce numerically similar vectors.

Concept
The database is a geometric space
After ingestion, every review exists as a point in high-dimensional space. Reviews that mean similar things cluster together — not because they share words, but because the model learned to place similar meanings close together from billions of training examples. Hover any point to explore.

Why clusters form
The embedding model was trained on massive text corpora. It learned that "salty," "puckering," "too much sodium," and "needed water" all appear in similar contexts — so they end up encoded near each other in the learned space.
Similarity metric
Distance is measured using cosine similarity — the angle between two vectors, not their absolute positions. This makes comparisons robust to text length. Score of 1.0 = identical meaning. 0.0 = unrelated.
Query time
Searching the vector space
When a search arrives, it goes through the same embedding pipeline as the stored reviews — then the database finds the nearest vectors. The query never touches the raw review text directly.

What just happened
  • 1
    Query embedded
    The search phrase is passed through the same embedding model used at ingestion time. This is critical — the same model must be used both times or the vectors won't be comparable.
  • 2
    ANN search
    The DB runs an Approximate Nearest Neighbour search — an index structure (HNSW, IVF) that avoids comparing against every stored vector. Millisecond latency even at millions of records.
  • 3
    Threshold applied
    Only vectors above the similarity threshold are returned. The cutoff is tunable — a legal doc system might require 0.92+, a brainstorming tool might accept 0.50+.
  • 4
    Results ranked
    Matches are returned ordered by similarity score. The original review text is fetched from the DB alongside the vector, so the user sees readable results.
